# PRD
## 目标：关键需求描述准确，减少无效沟通

## 首先思考问题：
1. 解决用户什么问题？发现问题
提案 -> 为什么要实现这个提案？ -> 因为用户.... -> 调查事实，为什么用户... -> 因为。。。 -> 假设改善。。。有助于...
-> 验证假设 -> 设计产品
2. 如何衡量问题是否 在不同用户组中被 解决？
设计什么衡量指标？有无埋点？有无业务数据？谁来运营？
3. 为了实现这个 设计产品，需要多少资源，人力？能评估吗？
4. 是不是太复杂，有没有性价比更高的 设计？会不会影响现有业务？会不会影响历史数据？
5. 有风险吗？违法？国家监管政策？
6. 竞品分析，其他类似的产品是怎么做的？
7. 用户对这个 设计产品 的看法？需求都满足了吗？有跟用户交流吗？有去实际体验场景吗？相关领域的知识？

## 任何一个问题没清晰之前，不要进行下一步。

## 然后动手设计：
1. 首先遍历出所有用户角色；
2. 针对每一个角色，提供对应的系统/功能（具备：输入、处理、输出 特性）系统(user center)> func module(user management, character management) > func(user list, add user,, band user, character list, add character) 
3. 把各个角色和功能联系起来，通过核心业务流程，宏观认识，得到子系统核心业务流程图，活动图，状态机图，与外部系统交互的时序图，总结产品提供的功能
4. 按照某种维度对功能进行划分，得到功能结构图
5. 墨刀绘制原型图，产品的界面和交互。用户输入指令后，输入哪些数据，表结构如何设计，系统根据逻辑处理，然后输出结果

## 产品设计文档部分
1. 记录每次文档更新的时间、作者、内容，历史追溯
2. 全局说明
包括名词的统一解释-便于沟通、异常处理逻辑、列表默认数据规则-排序方式、默认显示条数、翻页规则、缺失值规则等等。

## 产品背景，评价指标
1. 现状：需求方的问题，如何和我们的产品的核心价值挂钩（我们的差异化优势）
2. 方案：针对这个问题，我们的产品提供的解决方案
3. 目标：指标，我们的产品能期望获得多少的价值提升

核心业务流程 + 系统业务流程

功能结构图

## 功能需求描述
1. 对前面的功能结构图中的每个功能点逐一解释描述。现在不仅仅是解释功能，还要从用户角度进行描述。如：用胡添加好友、删除好友、查看列表：
2. 前置条件（用户要操作此功能，需要有什么角色属性、权限、状态）、
3. 后置条件（执行完这个功能之后，数据会发生什么样的变化，页面如何跳转）、
4. 界面交互（页面可以被拆分为多个元素，如表单、文本、链接、图片）表单元素应该要有详细描述，如这个空是否可以为空，是否要有初始内容，字数限制。。。链接跳转到哪个页面，图片显示比例，未加载成功显示什么，页面内容是写死还是通过后台动态配置
5. 业务流程（用户输入sumit, 后台怎么校验，怎么对不同输入不同操作处理，不同结果返回，通过业务流程图+文字描述）、
6. 异常和分支（网络错误、接口返回异常、服务器内部错误）、
7. 数据字典，数据库知识，数据库结构+算法，用户使用产品时是在与数据交互，只是在此之间，增加了一些算法对数据进行包装和运输

## non-functional
1. 数据埋点
2. 监控需求：监控某个接口/services，当异常发生时，可以报送异常信息至相关人员
3. 性能：scalability, security...
