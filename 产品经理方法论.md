21. WBS - **项目管理工具**
	1. 将复杂的项目目标逐步拆解为具体的可交付成果（Deliverables）和任务包（Work Packages），最终形成一个树状或列表形式的可视化结构。
	2. 确定项目范围的“基准文档”，任何变更需要对比WBS进行评审
	3. 防止范围蔓延：成员仅仅执行WBS定义的任务
	4. 用WBS作为共同语言，减少对人物理解的偏差
	5. 任何范围的变更需更新WBS并重新验证完整性
	6. 验收标准对齐：每个任务包定义明确的交付成果，便于阶段验收（如完成UI设计稿后进入开发）
	7. 以成果为导向：聚焦可交付成果，而非行为本身
	8. 层次化分解：从项目总体目标到具体任务，逐渐细化，如项目→阶段→任务→子任务
	9. 团队共识工具：确保所有参与者对项目任务分配范围的理解一致
	10. 创建步骤
		1. 明确项目目标
		2. 识别主要可交付成果
		3. 逐渐分解
			1. 按阶段分解（需求分析、设计、开发、测试、交付）
			2. 按功能模块分解（UI，数据库，后端）
			3. 按地理区域分解（建筑项目不同施工区域)
			4. 颗粒度过细会增加管理成本
			5. 聚焦做什么而非如何做
			6. 确保分解过程中获得关键干系人的认可
		4. 验证完整性
		5. 编码与可视化（任务编码，生成树状图/甘特图）
22. 因果图法 - 描述**输入条件的组合**以及每种组合对应的**输出**的图形化工具
	1. 测试法
	2. 如果程序输入之间没有什么联系，采用等价类划分和边界值分析是一种比较有效的方法
	3. 如果有关系，如约束关系、组合关系，这种关系用等价划分和边界值分析是很难描述的，测试结果难以保证，因此必须考虑一种适合于描述多种条件的组合，产生多个相应动作的测试方法。因果发着重测试规格说明中的输入和输出之间的依赖关系。
	4. 基本符号![[Pasted image 20251120162116.png]]
	5. 限制关系图形![[Pasted image 20251120162207.png]]
	6. 测试步骤
		1. 分析程序的规格说明书中哪些是输入条件/输入等价条件，哪些是输出条件
		2. 给每一个输入条件和输出条件赋一个标识符
		3. 分析输入条件与输入条件，输出条件与输出条件，输入条件与输出条件之间的关系，画出因果图
		4. 特殊情况无法画出因果图，用一些记号标明约束/限制条件
		5. 将因果图转换为判定表
		6. 根据判定表的每一列设计测试用例
	7. 通过将规格说明转换为布尔逻辑网络，就可以指出规格说明的不完整和不明确之处。
	8. 因果图法最具难度的部分是将因果图转换为判定表。但这个过程是有算法的，即意味着我们可以编写程序来自动完成这个过程。

24. **转换率**
	1. 从一个阶段进入下一个阶段的比例（用户比例）
	2. 达成目标用户/进入该环节的用户数
	3. e.g. 进入注册页：1000； 填写手机号：700；成功注册：500. 手机号填写转换率：70%，注册成功率：500/700
	4. 可以帮助判断用户卡在哪个节点，然后结合埋点数据分析原因，提出优化方向

39. 产品开发生命周期
	1. brainstorm
		1. 头脑风暴的团队成员多样性很重要
		2. 竞品分析
			1. 填补市场空白
			2. 比现有产品更好地解决问题
	2. define
		1. UX设计员、UX研究员、项目经理、产品负责人
		2. 通过回答
			1. 产品是为谁谁准备的？
			2. 这个产品能做什么？
			3. 需要哪些功能？
		3. 缩小想法焦点
		4. 在没有询问用户的情况下，不能直接假设知道用户正在经历什么
	3. design
		1. UX 设计
	4. test
		1. UX设计师和工程师一起开发符合原始设计的功能模型：Including适合公司的细节和功能
		2. 编写代码
		3. 确定产品整体结构
		4. 公司内部测试 alpha测试
		5. 利益相关者审查 公司愿景、合法、隐私法规
		6. 潜在用户的外部测试 beta测试
	5. launch
		1. 推广产品
		2. 营销专业人员可能会在社交媒体上发布有关新产品的信息或发布新闻稿
		3. 客户支持团队可能已经准备好帮助新用户了解产品的工作方式。
		4. 项目经理还会与跨职能团队会面，反思整个产品开发生命周期，并提出以下问题：哪些有效？哪些可以改进？目标实现了吗？是否符合时间表？为这种反思腾出时间非常重要，因为它可以帮助改进产品。
